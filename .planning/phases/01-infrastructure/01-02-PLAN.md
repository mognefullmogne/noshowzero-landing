<task type="auto">
  <name>Task 1: Create verification script with RLS auto-fix</name>
  <files>scripts/verify-infrastructure.mjs</files>
  <action>
Create `scripts/verify-infrastructure.mjs` -- a Node.js script that verifies all post-migration infrastructure state and auto-fixes known issues. Uses the same pattern as `run-migrations.mjs` (pg client, DATABASE_URL env var, SSL).

The script must perform these checks IN ORDER, printing pass/fail for each:

**1. Table existence check (INFRA-02)**
For each of the 16 tables from migrations 004-011, query `pg_tables` to confirm existence in the public schema:
```
message_threads, message_events, delivery_statuses,
appointment_slots, optimization_decisions,
rulesets, rule_versions, audit_events,
confirmation_workflows, slot_proposals, kpi_snapshots, failed_jobs,
booking_sessions, tenant_phone_numbers,
calendar_integrations, import_logs
```
Print: "PASS table_name exists" or "FAIL table_name missing"

**2. RLS enablement check (INFRA-04)**
For each of the 16 tables, query `pg_class.relrowsecurity` to confirm RLS is enabled.
Print: "PASS table_name has RLS enabled" or "FAIL table_name RLS disabled"

**3. RLS policy check (INFRA-04)**
For each of the 16 tables, query `pg_policies` to confirm at least one SELECT-compatible policy exists (cmd = 'r' for SELECT or cmd = '*' for ALL).
Special case for `audit_events`: accept SELECT-only policy as valid (it is insert-only via service_role by design -- see RESEARCH.md Pitfall 6).
Print: "PASS table_name has N policies" or "FAIL table_name has no SELECT policy"

**4. RLS policy auto-fix for migration 010 tables**
For `booking_sessions` and `tenant_phone_numbers`:
- Query `pg_policies` to check if any policy contains `current_setting` in the definition (query `pg_policy.polqual` via `pg_get_expr`)
- If found, execute corrective SQL:
  ```sql
  DROP POLICY IF EXISTS booking_sessions_tenant_isolation ON booking_sessions;
  CREATE POLICY "Tenant isolation" ON booking_sessions
    FOR ALL USING (tenant_id IN (SELECT id FROM tenants WHERE auth_user_id = auth.uid()));
  ```
  Same pattern for `tenant_phone_numbers`.
- Print: "FIXED booking_sessions RLS policy (was current_setting, now auth.uid())" or "OK booking_sessions RLS policy already correct"

Alternative approach for detecting the mismatch (simpler): query `pg_policies` for the table and check if policyname matches `booking_sessions_tenant_isolation` or `tenant_phone_numbers_tenant_isolation` (the migration 010 names). If those names exist, drop and recreate with the correct pattern. If policyname is "Tenant isolation" (the corrected name), skip.

**5. For any table missing RLS policies entirely** (not just migration 010 tables):
Auto-create a tenant-scoped policy matching the project pattern:
```sql
CREATE POLICY "Tenant isolation" ON {table_name}
  FOR ALL USING (tenant_id IN (SELECT id FROM tenants WHERE auth_user_id = auth.uid()));
```
Print: "CREATED missing RLS policy for table_name"

**6. Publication check (INFRA-03)**
Query `pg_publication_tables` to confirm `appointments` is in `supabase_realtime`.
Print: "PASS appointments in supabase_realtime publication" or "FAIL appointments not in publication"

**7. Replica identity check (INFRA-03)**
Query `pg_class.relreplident` to confirm `appointments` has `f` (FULL).
Print: "PASS appointments REPLICA IDENTITY = FULL" or "FAIL appointments REPLICA IDENTITY = {actual}"

**8. Data integrity check (INFRA-02)**
Query `patients` table for a record with last_name ILIKE '%rossi%'.
Query `appointments` table for a record with service_name ILIKE '%prostata%'.
Print: "PASS patient stefano rossi exists" or "FAIL patient record missing"
Print: "PASS appointment esame prostata exists" or "FAIL appointment record missing"

**9. Summary**
Count total PASS/FAIL/FIXED results. Print summary:
```
=== INFRASTRUCTURE VERIFICATION SUMMARY ===
PASS: N checks
FAIL: M checks
FIXED: K issues auto-corrected
```
If any FAIL, exit with code 1. If all PASS (with or without FIXED), exit with code 0.

**Script structure:**
- Use same `pg.Client` pattern as `run-migrations.mjs`
- Same `DATABASE_URL` env var requirement with clear usage instructions
- Wrap all queries in try/catch -- a query failure on one check should not abort remaining checks
- Collect results in an array: `{ check: string, status: 'PASS' | 'FAIL' | 'FIXED', detail: string }`
- Print results as they run, then summary at end
