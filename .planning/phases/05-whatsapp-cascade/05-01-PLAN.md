**trigger-backfill.ts** — Modify the orchestrator:
1. Add a `MAX_OFFERS_PER_SLOT` constant (set to 10 — reasonable cap per user's discretion). Before sending an offer, count existing offers for this slot: `SELECT count(*) FROM waitlist_offers WHERE original_appointment_id = ? AND status IN ('pending','accepted','declined','expired')`. If count >= MAX_OFFERS_PER_SLOT, log and return null without sending.
2. When `candidates.length === 0` (no candidates found), check if any offers were previously sent for this slot. If yes (offers exist but no candidates left), this is cascade exhaustion — send a staff notification. Use `console.warn("[Backfill] CASCADE EXHAUSTED: No more candidates for appointment:", appointmentId)` and create an audit_log entry: `{ tenant_id, actor_type: 'system', entity_type: 'appointment', entity_id: appointmentId, action: 'cascade_exhausted', metadata: { offers_sent: count } }`. (In-app notification via audit_log table which the dashboard already queries — Claude's discretion on notification channel per CONTEXT.md).
3. Remove the retry logic that tries `candidates[1]` on failure — the cascade handles retries via decline/expire. Instead, on send failure, log and return null (the next expiry/decline will cascade to the next candidate).

2. Keep the cascade call to `triggerBackfill` after marking the offer as expired — this is the core cascade mechanism.

**process-response.ts** — Clean up dead code in both functions:

3. Keep the `processDecline` cascade call to `triggerBackfill` — this is what makes decline->next-candidate work.
